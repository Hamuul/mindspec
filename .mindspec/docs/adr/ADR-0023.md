# ADR-0023: Eliminate Focus File — Beads as Single State Authority

- **Date**: 2026-03-03
- **Status**: Proposed
- **Domain(s)**: workflow, git, state
- **Deciders**: Max
- **Supersedes**: ADR-0022 §4 invariant 5 ("focus on main, updated on every transition"), ADR-0015 §state.json-as-cursor
- **Related**: ADR-0020 (per-spec lifecycle file), ADR-0022 (worktree-aware resolution)

---

## Context

MindSpec currently maintains `.mindspec/focus` as a JSON file tracking the active spec, bead, mode, worktree, and branch. ADR-0022 §4 invariant 5 says focus lives on main and is updated on every transition. In practice the code already drifted — `approve/spec.go`, `approve/plan.go`, and `specinit` write focus to the worktree, not main. The LLM test harness exposed this when assertions reading focus from main found stale or missing state.

Focus on main is fundamentally broken for parallel agents. If Agent A implements spec-042 and Agent B implements spec-043, both writing focus to main creates a last-writer-wins race. Per-worktree focus solves contention but adds a new problem: when an agent starts on main (fresh session), there's no focus file to read.

Meanwhile, **every field in the focus file is already derivable from beads + naming conventions**:

| Focus field | Beads / convention equivalent |
|:------------|:-----------------------------|
| `mode` | Derived from epic/bead statuses |
| `activeSpec` | Epic title: `[SPEC NNN-slug]` → spec ID |
| `activeBead` | `bd list --status=in_progress --parent <epicID>` |
| `specBranch` | Convention: `spec/<specID>` |
| `activeWorktree` | Convention: `.worktrees/worktree-<beadID>` or `.worktrees/worktree-spec-<specID>` |
| `timestamp` | Bead `updated_at` |

Focus is a denormalized cache of state that lives authoritatively in beads. The cache adds maintenance cost (every transition must write it), creates consistency bugs (stale/missing focus), and cannot represent parallel work.

Beads runs a Dolt server daemon (MySQL-compatible) with sub-millisecond local socket queries. The `.beads/` directory is shared across all worktrees via git's common directory mechanism. Beads already tracks:

- Which specs exist (epics with `[SPEC ...]` title convention)
- Which phase each spec is in (derivable from epic/bead statuses)
- Which beads are open, in_progress, closed, or blocked
- Dependency relationships between beads
- Cross-worktree state without requiring any worktree to be checked out

## Decision

### 1. Eliminate the focus file

No `.mindspec/focus` file anywhere — not on main, not in worktrees. All state queries go through beads.

CLI commands that currently write focus (`approve spec`, `approve plan`, `next`, `complete`, `specinit`) stop writing it. Commands that currently read focus derive the same information from beads instead.

### 2. Lifecycle discovery and phase derivation via beads

All lifecycle queries use beads as the single source of truth:

**Active spec discovery** (replaces `ActiveSpecs()` filesystem scan):
```
bd list --type=epic --status=open --json
```

**Phase derivation** from epic/bead statuses:

| Epic state | Derived phase |
|:-----------|:-------------|
| Epic open, no children | spec |
| Epic open, all children open (none claimed) | plan |
| Epic open, a child is in_progress | implement |
| Epic open, all children closed | review |
| Epic closed | done |

**Active bead** (replaces `focus.activeBead`):
```
bd list --status=in_progress --parent <epicID> --json
```

**Blocked bead detection** (replaces focus-based checks):
```
bd blocked --parent <epicID> --json
```

This replaces filesystem scanning of `lifecycle.yaml` across worktrees. It works regardless of whether worktrees exist on disk, which branch the agent is on, or whether lifecycle.yaml has been merged to main.

### 3. Worktree-to-spec resolution by convention

Worktree paths are deterministic from IDs — no focus file needed to record them:

| Context | Path convention |
|:--------|:---------------|
| Spec worktree | `.worktrees/worktree-spec-<specID>` |
| Bead worktree | `.worktrees/worktree-<beadID>` |
| Spec branch | `spec/<specID>` |
| Bead branch | `bead/<beadID>` |

When an agent starts on main, `mindspec instruct` queries beads for open epics, derives the phase, and determines which worktree(s) exist by convention. It then emits the CWD redirect.

When an agent is already in a worktree, the worktree name encodes the spec or bead ID. The CLI extracts it from the path or branch name.

### 4. Beads as the single cross-worktree state store

| Concern | Source of truth | Location |
|:--------|:---------------|:---------|
| Which specs are active | Beads epics (open) | `.beads/` (Dolt, shared) |
| Spec lifecycle phase | Derived from bead statuses | `.beads/` (Dolt, shared) |
| Which bead is claimed | Bead status = in_progress | `.beads/` (Dolt, shared) |
| Bead dependencies/blocking | Beads dependency graph | `.beads/` (Dolt, shared) |
| Spec artifacts | Worktree branch files | Per-worktree filesystem |
| Lifecycle phase marker | lifecycle.yaml | Per-worktree filesystem |

`lifecycle.yaml` remains as a per-worktree convenience file for spec artifacts (co-located with spec.md/plan.md on the branch). But it is not consulted for cross-worktree discovery — beads is.

### 5. Updated invariants (supersedes ADR-0022 §4)

1. **Main is read-only** — no direct commits, no file writes. Only receives merges. *(unchanged)*
2. **One worktree per unit of work** — specs, beads, and bugfixes each get their own worktree. *(unchanged)*
3. **Worktrees are disposable** — created by `mindspec next`/`spec-init`, removed by `mindspec complete`/`cleanup`. *(unchanged)*
4. **Context switch = worktree switch** — each worktree has its own filesystem state. *(unchanged)*
5. **No focus file** — active context is derived from beads + path conventions, not cached in a file. *(changed — was "focus on main")*
6. **Beads is the single cross-worktree state store** — lifecycle discovery, phase derivation, and work-readiness queries go through beads. *(new)*

## Consequences

### Positive

- **Parallel agents work without contention** — no shared mutable state file; beads handles concurrent reads/writes via Dolt
- **Single source of truth** — eliminates the consistency bug class where focus disagrees with beads (stale mode, wrong bead, missing file)
- **Lifecycle discovery works from any context** — beads is always available via the Dolt daemon, no worktree checkout or filesystem scan required
- **Less code** — remove all `WriteFocus`/`ReadFocus` calls and the `state.Focus` struct; remove `assertFocusMode`/`assertFocusFields` test helpers in favor of beads assertions
- **Aligns with ADR-0020** — lifecycle.yaml is a per-spec worktree artifact, beads provides the cross-spec operational view

### Negative / Tradeoffs

- **Beads daemon must be running** — if the Dolt server is down, no state queries work. Mitigated by: beads auto-starts on first `bd` call; `mindspec doctor` checks daemon health; the daemon is already required for all lifecycle operations.
- **Epic title convention is load-bearing** — `[SPEC NNN-slug]` prefix is how discovery maps epics to specs. Mitigated by: `plan approve` enforces this convention; a structured field (e.g. `spec_id` tag on the epic) could replace string parsing later.
- **Slightly more beads queries per command** — commands that previously read a local JSON file now query Dolt. Mitigated by: Dolt daemon queries over local socket are sub-millisecond; the total overhead is negligible.
- **Migration** — existing repos have focus files that must be cleaned up. Mitigated by: `mindspec doctor` can detect and remove stale focus files; focus removal is backward-compatible (commands that can't find focus already fall back to beads/lifecycle scanning).

## Alternatives Considered

### 1. Per-worktree focus files (no main focus)

Keep focus files but only in worktrees, not on main.

- Pros: Faster local reads; familiar pattern
- Cons: Still a denormalized cache that can drift from beads; still must be written on every transition; adds no information that beads doesn't already have
- Rejected: if beads is always authoritative, the cache provides no value and creates consistency risk.

### 2. Focus on main with agent-ID namespacing

Multiple focus entries on main, keyed by agent ID.

- Pros: Single location; no beads dependency for reads
- Cons: Schema change; requires agent identity; merge conflicts on parallel writes; still a projection of beads state
- Rejected: complex solution to a problem that goes away by reading beads directly.

### 3. Keep focus as a performance cache with beads as fallback

Read focus first, fall back to beads if focus is missing/stale.

- Pros: Fast path for the common case
- Cons: Two code paths to maintain; "which is authoritative?" confusion; stale cache bugs are the current problem
- Rejected: the performance gain is negligible (sub-ms Dolt queries) and the consistency cost is high.

## Validation / Rollout

1. Add `DiscoverActiveSpecs()` function that queries `bd list --type=epic --status=open --json` and derives phase from bead statuses
2. Add `ResolveContext(root)` function that returns `(specID, beadID, phase, worktreePath)` by combining beads query with path conventions
3. Update `instruct` to use `ResolveContext()` instead of `ReadFocus()`
4. Update `next`, `complete`, `approve` commands to derive context from beads instead of reading/writing focus
5. Remove `state.Focus` struct, `ReadFocus()`, `WriteFocus()`, and all callers
6. Update LLM test harness: replace `assertFocusMode`/`assertFocusFields` with beads-based assertions (`assertBeadsState`, phase derivation checks)
7. Update `sandbox.WriteFocus()` in test setups to use `bd update` for setting bead status instead
8. `mindspec doctor` detects and removes stale `.mindspec/focus` files
9. Run full LLM test suite to verify
